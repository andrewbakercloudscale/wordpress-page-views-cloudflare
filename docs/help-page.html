Statistics Dashboard

Why CloudScale counts traffic that Jetpack and WP Stats miss

When Cloudflare, WP Rocket, or any CDN serves a cached HTML page, WordPress never executes — so server-side analytics miss the view entirely. On typical cached sites this means 80–95% of real visits are invisible to your stats dashboard.

CloudScale solves this with a lightweight JavaScript beacon that fires after the browser loads the page. It POSTs to a REST endpoint that bypasses the CDN cache, so every real visit is counted regardless of whether the HTML came from cache or from PHP.

All data is stored in your own WordPress database. No third-party tracking scripts, no Google Analytics, no cookies required, no GDPR consent banner needed.

The Statistics Dashboard is the main view under Tools → CloudScale Site Analytics. It reads directly from five database tables: wp_cspv_views_v2, wp_cspv_referrers_v2, wp_cspv_geo_v2, wp_cspv_visitors_v2, and wp_cspv_404_v2.

Date Range & Stat Cards

  • Period selector — Today, 7 days, 30 days, 90 days, or All Time. Each period queries wp_cspv_views_v2 directly; no aggregation tables needed.
  • Stat cards — Total Views, Unique Visitors, and Unique Posts for the selected period, with percentage change versus the prior equivalent period (e.g. previous 7 days when viewing the last 7 days).
  • All Time Views — lifetime total sourced from the beacon log (wp_cspv_views_v2), shown separately below the period cards.

Views Chart

Daily bar chart for the selected period. Gaps indicate days with zero traffic — not missing data. Hover any bar to see the exact date and view count. The chart updates instantly when you change the period selector.

Top Posts (Period)

Ranked table of posts by view count for the selected period. Each row links directly to the post's edit screen. Use this to identify which content is performing in any given window, then compare across periods by switching the selector.

Referrers

The referrer list shows where traffic is arriving from, parsed from the document.referrer value sent by each beacon. Stored in wp_cspv_referrers_v2. Toggle between Sites (referrer domain) and Pages (full referring URL).

Click any referrer domain to open the Referrer Drill-Down modal showing the top pages on that domain that linked to your site.

Cloudflare Cache Bypass Check

Without a Cloudflare Cache Rule, Cloudflare caches the beacon REST responses and POST requests fail silently — your view counts appear to record but nothing is written. The bypass check panel sends a test probe and confirms the rule is working.

Required rule: Cloudflare dashboard → Caching → Cache Rules → URI Path contains /wp-json/cloudscale-wordpress-free-analytics/ → Cache Status: Bypass.

404 Error Log

Every 404 response is logged with the requested URL, referrer, and timestamp (stored in wp_cspv_404_v2). Use this to find broken links from external sites, spot content that moved without a redirect, and set up 301 redirects for your most-hit missing paths.

Referrer Drill-Down

The Referrer Drill-Down lets you look inside any referring domain to see exactly which pages on that site are sending you traffic.

How to open it

In the Statistics Dashboard, click any domain name in the Referrers list. A modal opens showing:

  • Domain header — the referrer host and total hits from that domain for the selected period.
  • Top source pages — a ranked list of the specific URLs on that domain that linked to your site, with the hit count for each.
  • Page links — each source page URL is clickable so you can visit it directly to see the context of the link.

Pagination

If a domain has more than 10 source pages for the period, the modal paginates. Use the Prev / Next buttons to scroll through the full list.

Close the modal with the × button or by clicking outside it. The drill-down respects the active period selector — switching periods and re-opening any referrer will show updated data for the new date range.

Geography

An interactive world map showing where your visitors come from, powered by data in wp_cspv_geo_v2. Country-level data is captured automatically with no setup required — it works as soon as the plugin is active and Cloudflare is in front of your site.

How country detection works

Every page request that passes through Cloudflare carries a CF-IPCountry header containing the visitor's two-letter ISO country code. CloudScale reads this header on each beacon hit and stores the country code in wp_cspv_geo_v2 — no IP geolocation database needed, no raw IP stored.

If your site is not behind Cloudflare, the geography map will show no data — the plugin does not perform its own IP-to-country lookup by default.

Map features

  • World map — bubble markers sized by relative visit volume. Click any country bubble to drill down to a country-level breakdown.
  • Country list — ranked table of countries with visit counts for the active period, shown below the map.
  • Reset Map — clears any active drill-down and returns to the full world view.

The geography view respects the active date range — switching the period selector updates both the map and the country list.

Display Settings

The Display Settings tab controls the view counter badge on your posts, configures sidebar widgets, and provides data management tools.

View Counter

  • Position — before post content, after post content, both, or off. Requires no theme editing; the counter is injected via the the_content filter.
  • Style — Badge (coloured pill with gradient), Pill (outlined), or Minimal (plain text).
  • Colour — five options: blue, pink, red, purple, grey.
  • Post types — comma-separated list of post types to exclude from tracking (e.g. product for WooCommerce). Excluded types do not fire the beacon and are not counted.

Sidebar Widgets

  • Top Posts widget — register via Appearance → Widgets. Queries wp_cspv_views_v2 for a configurable view window. Settings: total posts to pool, posts per page, thumbnail width, sort order (most viewed or most recent).
  • Recent Posts widget — shows most recently published posts with live view counts. Configurable post count and optional date display.

Data Management

The Purge Visitor Hashes tool removes historical deduplication data from wp_cspv_visitors_v2 older than a selected threshold (30, 60, 90, 180 days, 1 year, or all data). The table grows indefinitely unless periodically pruned — the panel shows the current row count and date range before you act.

Purging only removes the hash records used for future deduplication. View counts already written to wp_cspv_views_v2 are not affected.

IP Throttle

The IP Throttle tab has four panels that protect your stats from bots, scrapers, repeat loads, and your own test browsing.

IP Throttle Protection

Automatically blocks IP addresses that send an excessive number of beacon requests within a rolling time window. Blocked IPs receive an HTTP 200 (silent drop) — no error signal for attackers to adapt to. Blocks auto-expire after 1 hour.

  • Block after — maximum requests per IP within the window before the IP is blocked.
  • Time window — rolling window: 10 min, 30 min, 1 hr, 2 hr, or 24 hr.
  • Exclude logged-in users — authenticated sessions are never throttled.
  • Exclude administrators — only the administrator role is excluded; editors and authors are still counted.

View Deduplication

Prevents the same visitor inflating counts by viewing the same post repeatedly within a configurable window. Works at two levels:

  • Client-side (localStorage) — beacon records a key in localStorage on first fire. Subsequent visits from the same browser within the window do not fire again.
  • Server-side (IP hash + post ID) — the REST endpoint checks wp_cspv_views_v2 for a recent row matching the same hashed IP and post within the dedup window, catching clients that clear localStorage or use private browsing.

Configure the dedup window from 1 hour to 48 hours to balance accurate unique-visitor counting against legitimate repeat reads.

Blocked IPs

Lists all currently blocked IP hashes with block timestamps and time-to-expiry. IPs are stored as one-way SHA-256 hashes — not reversible to a real IP.

  • Unblock — removes a specific hash immediately, before auto-expiry.
  • Clear All — removes all blocked IPs at once. Use after false-positives from load tests.

Page Tracking (Kill Switch)

An emergency toggle that instantly stops all view tracking site-wide. When paused, the beacon script is not loaded on any page and the REST endpoint silently rejects requests. All historical data is preserved. The panel header badge shows TRACKING ACTIVE (green) or TRACKING PAUSED (red) at a glance.

Post History

The Post History tab lets you search any post or page and see a searchable list of all posts with beacon data, including lifetime view totals and trending direction.

Search & List

  • Search bar — live-filter the post list by title. Supports posts, pages, and any tracked custom post type.
  • List columns — Post title (with edit link), first recorded view date, last recorded view date, and total beacon hits.
  • Sort — click any column header to sort ascending or descending.

Row Expand: 30-Day Day-by-Day Breakdown

Click any row to expand it in place. The expansion shows the last 30 days that had at least one view for that post — days with zero views are hidden to keep the list concise.

Each expanded row shows:

  • Date — the calendar date.
  • Views — total beacon hits for that day.
  • Traffic bar — a mini proportional bar chart split by referral source:
    • Green segment — visits from your own site (self-referrals, e.g. internal links). Labelled self: N.
    • Blue segment — visits from an external referrer. Labelled with the referring domain.
    • When both self and external referrals exist on the same day, both segments appear side by side in the same bar row.

Click the same row again to collapse the expansion. Opening a different row automatically collapses any previously open row.

Trending Indicator

Each row includes a trending badge comparing the last 7 days against the post's 28-day average daily rate. Green arrow means traffic is accelerating; red arrow means it is declining. Posts with fewer than 7 days of data show "Insufficient history".
